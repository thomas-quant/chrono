# Phase 4 — UI Design Contract

> Visual and interaction contract for the QR/Barcode scan-to-dismiss alarm task. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Platform:** Flutter / Android, Material 3 (Material You). This is NOT a web/React/shadcn project — the shadcn/registry rows below are marked *not applicable — Flutter Material app*. All tokens are extracted from the EXISTING Chrono design language (`lib/theme/`, `lib/alarm/widgets/tasks/`, `lib/common/widgets/fields/`), not invented. Express color as Material 3 *roles*, never hex — the app supports Material You dynamic color + user-selectable color schemes.
>
> **Scope of this contract:** the five concrete UI surfaces the locked CONTEXT decisions require — (1) inline registration card, (2) escape-hatch toggle, (3) ring-time scanner widget, (4) "unlock to scan" keyguard prompt, (5) accessibility/localization cross-cuts. Almost everything is *assembly of existing seams*; this contract pins the visual/interaction details so the executor has no design ambiguity.

## Spacing Scale

Declared values (all multiples of 4, matching the spacing already used across Chrono's task widgets, settings cards, and switch fields). `EdgeInsets.all(16.0)` is the dominant container padding; `4` is the gap unit; `12` appears as vertical row padding inside fields (`switch_field.dart:37`).

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px | Gap between a label and its description/status line (`SizedBox(height: 4)`); icon-to-text inline gaps |
| sm | 8px | Compact spacing between stacked controls inside the scanner overlay |
| (field-row) | 12px | Vertical padding inside the escape-hatch `SwitchField` row (inherited from `SwitchField`; mirrors `switch_field.dart:37`) |
| md | 16px | **Default.** Card padding (`CardContainer` inner `EdgeInsets.all(16)`); registration card & task-widget root padding; vertical rhythm between the scanner instruction header and the preview |
| lg | 24px | Spacing around the over-keyguard "unlock to scan" prompt block; breathing room above the ring-time escape "Dismiss" button when it appears |
| xl | 32px | Reserved (large layout gap) — not expected to be needed in this phase |

**Exceptions:**
- **Touch targets:** all tappable controls (torch toggle, "Scan to register" card, escape "Dismiss" button, keyguard "Unlock" action) MUST have a minimum 48×48dp hit area (Material `IconButton`/`InkWell` defaults satisfy this; do not shrink below it). This is an accessibility floor, not a visual size.
- The scan-frame overlay reticle is a centered, square, semi-transparent cutout — its size is proportional to the preview (table-stakes; `flutter_zxing` `ReaderWidget` ships a default overlay). `[DEFAULT — confirm]` use the `ReaderWidget` built-in `scannerOverlay` rather than a custom one unless the executor finds it visually inconsistent.

## Typography

Material 3 text roles from `lib/theme/text.dart`, rendered in **Rubik** (variable font; weights via `FontVariations`). This phase uses exactly the roles already established for task widgets and settings cards — it introduces NO new sizes or weights.

| Role (this phase usage) | Material 3 role | Size | Weight | Where used |
|-------------------------|-----------------|------|--------|------------|
| Task / scanner instruction header | `headlineMedium` | 16px | semiBold (600) | "Scan the registered code" header in the ring-time `ScanTask` (mirrors `math_task.dart:142` "Solve the equation" → `textTheme.headlineMedium`) |
| Setting/card title | `displaySmall` | 16px | bold (700) | Registration card title "Scan code to dismiss" / "Registered code" (mirrors `setting_action_card.dart:42`) |
| Body / description / status | `bodyMedium` | 12px | regular (400) | "✓ Code registered" / "Not registered yet" status line; escape-hatch toggle description; "unlock to scan" explanatory copy (mirrors card description style, `setting_action_card.dart:48`) |
| Switch field name | `headlineMedium` | 16px | semiBold (600) | Escape-hatch toggle label (mirrors `switch_field.dart:43`) |

**Line heights:** inherit Material 3 / Rubik defaults from the theme — do NOT override per-widget (consistent with every existing task widget, none of which set explicit `height`). Body copy reads at the theme default (~1.4–1.5); headers at the theme default (~1.2). No custom line-height tokens introduced.

**Weight discipline:** the app's `FontVariations` palette is wide, but this phase uses only the weights its mirrored widgets already use — **regular (400)** for body/status and **semiBold (600)** for instruction headers / switch labels (bold/700 only where the existing `displaySmall` card-title role already applies it). No new weight is introduced.

## Color

All colors are **Material 3 `ColorScheme` roles**, resolved at runtime from the active `ColorSchemeData` (user-selectable scheme + Material You dynamic color). NEVER hardcode hex. The role mapping (from `lib/theme/types/color_scheme.dart:118`) is: user "Accent" → `primary` **and** `secondary`; user "Card" → `surface`; user "Background" → `background`; user "Error" → `error`; "Text" colors → `onBackground`/`onSurface`/`onError`.

| Role | Material 3 token | Usage in this phase |
|------|------------------|---------------------|
| Dominant (60%) | `background` / `onBackground` | Ring screen and settings backdrop; the camera *preview* itself is the dominant surface at ring time (live video fills the slot) |
| Secondary (30%) | `surface` (`card`) + `onSurface` | The registration card, the escape-hatch toggle row container, the scanner's control/instruction chrome rendered over/around the preview |
| Accent (10%) | `primary` (= `secondary` = user accent) | RESERVED — see explicit list below |
| Destructive / negative feedback | `error` / `onError` | The brief "not the registered code" wrong-scan feedback flash + the wrong-scan banner/icon tint |

**Accent (`primary`) reserved for — explicit list (never "all interactive elements"):**
- The registration card's "Scan to register" affordance state + the "✓ Code registered" success check icon.
- The `Switch` thumb/track ON state of the escape-hatch toggle (Material `Switch` already pulls `primary` from the theme — no override).
- The torch toggle's **active/ON** state in the scanner (off = neutral `onSurface`/outline; on = `primary` tint). Torch default state is OFF (SCAN-09).
- The scan-frame overlay reticle accent / "aim here" affordance, if the chosen overlay tints it.
- The ring-time escape-hatch "Dismiss" button (its accent prominence is acceptable here because it is a deliberate, ethics-critical safety affordance, not decoration).

**Destructive / error role usage (this phase has NO data-destroying action — see Copywriting):**
- `error` is used ONLY for the **wrong-scan feedback** (D-RING-WRONGSCAN): a brief visual flash/border/icon in `error` + a haptic buzz, signalling "not the registered code." It is transient feedback, not a destructive confirmation. `[DEFAULT — confirm]` wrong-scan visual = a short (~600ms) `error`-tinted overlay border + inline message, paired with `vibration` haptic; tune duration on-device so it never strobes.

Other locales follow via Weblate (English baseline only). **NEVER display or log the raw decoded code value** (D-REG-DISPLAY / privacy) — status only.

| Element | Copy | Proposed ARB key |

| Task name (in "Choose Task to Add" list, alongside Math/Retype/Sequence/Memory) | `Scan code to dismiss` | `scanTask` |
| Registration card title | `Registered code` | `scanRegisteredCodeTitle` |
| Primary CTA — register (no code yet) | `Scan to register` | `scanRegisterButton` |
| Primary CTA — re-test / replace (code already registered) | `Scan again to re-test` | `scanRescanButton` |
| Registration status — registered | `✓ Code registered` | `scanCodeRegistered` |
| Registration status — empty / required-to-save | `No code registered yet` | `scanNoCodeRegistered` |
| Save-blocked validation (D-REG-REQUIRED — registered code required to save the task) | `Register a code before saving this task` | `scanCodeRequired` |
| Escape-hatch toggle label | `Escape hatch` | `scanEscapeHatch` |
| Escape-hatch toggle description (the threshold lives behind the toggle — on/off only in v1) | `Lets you dismiss the alarm if scanning fails or the camera can't be used` | `scanEscapeHatchDescription` |
| Ring-time scanner instruction header | `Scan the registered code` | `scanRingInstruction` |
| Wrong-scan feedback (transient) | `That's not the registered code` | `scanWrongCode` |
| Torch toggle label (semantic / tooltip) | `Flashlight` | `scanTorchLabel` |

| Ring-time escape "Dismiss" button (when escape hatch fires) | `Dismiss` | reuse existing `dismissAlarmButton` (already in ARB, line 404 — do not add a duplicate) |
| Camera-denied at setup (D-REG-CAMDENIED — deep-link to system settings) | `Camera access is needed to scan a code. Open settings to allow it?` | `scanCameraPermissionPrompt` |
| Camera-denied — open-settings action | `Open settings` | `scanOpenSettings` |
| "Unlock to scan" keyguard prompt heading (D-LOCK-NOGO-UX) | `Unlock to scan` | `scanUnlockToScanTitle` |
| "Unlock to scan" keyguard prompt body (alarm keeps ringing until unlocked; escape hatch underneath) | `Unlock your device to open the scanner. The alarm keeps ringing until you dismiss it.` | `scanUnlockToScanBody` |

**Primary CTA for this phase:** `Scan to register` (verb + noun) — the registration action inside the inline card.

**Empty state:** there is no list/empty-state screen in this phase. The closest analog is the unregistered registration card: status reads `No code registered yet` + the `Scan to register` CTA. (The task itself appears in the existing "Choose Task to Add" list whose empty state `noTaskMessage` already exists — unchanged.)

**Error states:** (1) camera permission denied → `scanCameraPermissionPrompt` + `scanOpenSettings` deep-link, then resume. (2) camera unavailable at ring time → escape hatch fires INSTANTLY (SCAN-07), surfacing the `Dismiss` affordance — no error dialog blocks the user. (3) wrong code scanned → transient `scanWrongCode` + haptic, counts toward the escape threshold. (4) torch can't enable → `scanTorchUnavailable` toast, scanner keeps running.

**Destructive actions:** NONE in this phase. Deleting/duplicating a task uses the existing list-item slidable actions (`deleteButton`/`duplicateButton`, already in ARB) — out of this contract's scope. The scan task gates **dismiss only**; snooze stays a normal pre-task tap (SCAN-05 / D-RING-SNOOZE) and is never placed inside the scanner. No new confirmation dialog is introduced.

## Interaction Contracts (Flutter Material — per surface)

> The template's six checker dimensions are covered above (Copywriting, Typography, Color, Spacing) and below (Registry Safety). This section pins the *interaction* contract for each of the five locked surfaces so the executor and the ui-auditor have an unambiguous reference. All five derive directly from CONTEXT decisions.

### Surface 1 — Inline registration card (setup; D-REG-UI / D-REG-DISPLAY / D-REG-REQUIRED / D-REG-CAMDENIED)
- **Where:** rendered inside the scan task's `SettingGroup` page (`getSettingWidgets` over `customize_list_item_screen`), alongside the escape-hatch toggle. **Inline card, NOT a separate dedicated screen.**
- **Visual:** mirror `SettingActionCard` — `CardContainer` (surface, 16px padding) → `InkWell` → Row[ Column( title `displaySmall` "Registered code" + status `bodyMedium` ) , trailing affordance ]. Tapping runs the "Scan to register" action.
- **Status display:** STATUS ONLY. `✓ Code registered` (with a `primary`-tinted check) when set; `No code registered yet` when empty. **Never render the raw decoded value.**
- **Flow:** tap → `Permission.camera` request (`permission_handler`). Granted → push the registration scanner screen (mirror `try_alarm_task_screen.dart` Scaffold + `AppTopBar` + `ReaderWidget`); on a valid decode, normalize and store into the `StringSetting`, then pop with updated status. Denied → show `scanCameraPermissionPrompt` with `scanOpenSettings` → `AppSettings.openAppSettings()` deep-link, then resume.
- **Registration IS the test scan (D-REG-TEST / SCAN-10):** a successful register inherently proves it scans. When a code already exists, the CTA becomes `Scan again to re-test` (optional, non-mandatory; re-scanning REPLACES per D-REG-REQUIRED).
- **Save gate (D-REG-REQUIRED):** the task cannot be saved with no registered code → block with `scanCodeRequired` validation copy.

### Surface 2 — Escape-hatch toggle (setup; D-ESC-EXPOSURE / SCAN-06/07)
- **Visual:** a single `SwitchField` (label `scanEscapeHatch` `headlineMedium`, description `scanEscapeHatchDescription` `bodyMedium`, `Switch` thumb ON = `primary`). **DEFAULT ON.**
- **Exposure:** on/off ONLY — the threshold numbers (≈120s OR ~10 debounced attempts) live behind it. No fine-grained time/attempt knobs in v1 (those are v2 / SCAN-V2-02).
- **Accessibility (ethics-critical):** the toggle MUST be screen-reader reachable — it is the accessibility path. Spec: the `SwitchField`'s `InkWell` row is a single focusable node; expose a `Semantics` label equivalent to "Escape hatch, <on/off>, lets you dismiss the alarm if scanning fails or the camera can't be used." Do not hide the toggle behind a sub-screen.

### Surface 3 — Ring-time scanner widget (`ScanTask`; D-RING-LAYOUT / D-RING-SNOOZE / D-RING-WRONGSCAN / SCAN-09/11)
- **Where:** a normal task widget rendered FULL-SCREEN in the dismiss step (`_setNextWidget()` → `task.builder(onSolve)` in `alarm_notification_screen.dart`), entered via the existing swipe-to-dismiss `SlideNotificationAction`. NO new ring orchestration. Volume auto-lowers to `volumeDuringTasks` (inherited — no new UI).
- **Layout:** root `Padding(all: 16)` → `Column` with the `headlineMedium` instruction header `scanRingInstruction` at top (mirrors the math-task header), then the live `ReaderWidget` preview filling the slot, with the built-in scan-frame overlay reticle centered.
- **Torch toggle (SCAN-09):** use `ReaderWidget(showFlashlight: true)`; **default OFF**; OFF = neutral `onSurface`/outline tint, ON = `primary` tint; on failure show `scanTorchUnavailable` and keep scanning (graceful no-flash handling). Min 48dp hit area; `Semantics`/tooltip label `scanTorchLabel`.
- **Wrong-scan feedback (D-RING-WRONGSCAN):** a non-matching valid decode → brief (~600ms `[DEFAULT — confirm]`) `error`-role visual flash + `scanWrongCode` message + `vibration` haptic, and it counts toward the escape threshold (debounced — count distinct non-matching payloads / rely on `ReaderWidget.scanDelay` 1000ms; never count raw decode frames).
- **Match → dismiss:** valid decode where `codesMatch(normalize(text), stored)` → `onSolve()` (advances to next task or dismisses). The match is the ONLY non-escape success path.
- **Snooze:** NOT inside the scanner. Snooze remains the pre-task `SlideNotificationAction.onSnooze` swipe, reachable without ever entering the scanner (SCAN-05). Do not add any snooze affordance to the scanner.
- **Escape-hatch "Dismiss" affordance (SCAN-06/07):** when the `EscapeHatchController` fires (≥ time OR ≥ attempts, or instantly on camera-denied/unavailable), reveal a clearly-labelled `Dismiss` button (reuse `dismissAlarmButton`), `primary`-prominent, min 48dp, wrapped in `Semantics` so it is screen-reader reachable. Tapping calls `onSolve()` (skips only the scan task — D-ESC-SCOPE).
- **Camera release (SCAN-11):** every exit path (match→onSolve, escape→onSolve, app→background) must remove `ScanTask` from the tree so `ReaderWidget.dispose()` releases the `CameraController` — no stuck privacy indicator. (Defensive: also stop the controller on escape-fire via the `onControllerCreated` reference.)

### Surface 4 — "Unlock to scan" keyguard prompt (D-LOCK-NOGO-UX)
- **When:** shown over the secure keyguard ONLY on lock-screen-camera no-go devices (decided by the criterion-#1 spike). On go devices the scanner opens directly over the keyguard and this surface never appears.
- **Visual:** a centered block over the ring screen — heading `scanUnlockToScanTitle` (`displaySmall`/`headlineMedium`), body `scanUnlockToScanBody` (`bodyMedium`), 24px breathing room, on the `surface`/`background` backdrop consistent with the ring screen.
- **Behavior:** prompts the user to unlock; the scanner opens once unlocked; **the alarm keeps ringing until then.** The escape hatch is ALWAYS underneath (its threshold timer/attempts still apply, so the user can never be trapped even if they never unlock).

### Surface 5 — Accessibility & Localization (cross-cutting; SCAN-06/07/12 + CLAUDE.md ethics)
- **Screen-reader reachable:** the escape-hatch toggle (Surface 2) and the ring-time `Dismiss` button + torch toggle (Surface 3) MUST carry explicit `Semantics` labels and be focusable. The escape hatch is BOTH the safety path and the accessibility path — non-negotiable.
- **Large text / contrast:** inherit Material 3 theme scaling and the active `ColorScheme` contrast — do NOT hardcode font sizes or colors anywhere in this feature (use `textTheme.*` roles + `colorScheme.*` roles exclusively, as every mirrored widget does).
- **Localization (SCAN-12):** every string in the Copywriting Contract above is flagged for `lib/l10n/app_en.arb` (English baseline; Weblate for the other 20+ locales). The lone reused key is `dismissAlarmButton`. Run `flutter gen-l10n` (CI/human gate — toolchain absent locally) after editing the ARB.
