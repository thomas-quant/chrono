<objective>
Build the setup/registration half of the scan task: an inline "Scan to register" card (D-REG-UI) inside the scan task's `SettingGroup`, a registration scanner screen that opens only after a setup-time camera permission grant (SCAN-08), normalizes and stores the decoded code into the task's `StringSetting` (SCAN-02), shows status only ("Code registered" / "No code registered yet" — never the raw value, D-REG-DISPLAY), treats registration AS the test scan with an optional "Scan again to re-test" (D-REG-TEST / SCAN-10), deep-links to system settings on permission-denied then resumes (D-REG-CAMDENIED), enforces a REAL save gate so a scan task with no registered code cannot be saved (D-REG-REQUIRED), and removes the pre-existing `print(setting.value)` payload leak.

Purpose: This completes the user-facing setup flow so the ring-time task (Plan 04) has a code to match. Camera permission is requested at SETUP, never at fire time (SCAN-08). Implements SCAN-02, SCAN-08, SCAN-10. **The save gate (D-REG-REQUIRED) is enforced as an actual block on the task settings Save control** (UI-SPEC Surface 1 calls it a *save gate*, not just copy) — it is the whole point per CONTEXT.md: prevent shipping an alarm that can only ever be escape-hatched. All ARB keys already exist (Plan 04 owns app_en.arb, incl. `scanCodeRequired`) — this plan does NOT touch the ARB.

**Planner decision (PATTERNS O2 / D-STORE-FORMAT):** use route (B) — a plain `StringSetting "Registered Code"` (already added in Plan 04, `isVisual:false`) PLUS a marker setting that dispatches to a custom `ScanRegisterCard` in `get_setting_widget.dart`. This honors D-STORE-FORMAT literally ("no `json_serialize.dart` factory entry needed") — the `CustomSetting` route (A) would require a `fromJsonFactories[RegisteredCode]` entry, which D-STORE-FORMAT explicitly avoids. The raw value stays in the plain `StringSetting`; the card is a non-persisted view over it.

**Planner decision (save-gate seam, D-REG-REQUIRED):** the scan task's settings page is a `CustomizeListItemScreen<AlarmTask>` wrapped by `CustomizeScreen` (`customize_screen.dart`), whose `saveButton` `TextButton.onPressed` (lines 59-66) is the single confirm/save control. Enforce the gate there via a small, default-no-op validation hook on `CustomizableListItem` (`AlarmTask.validate()` returns the `scanCodeRequired` key only when `type == scan` and `normalizeCode(getSetting("Registered Code").value).isEmpty`), threaded `CustomizeListItemScreen → CustomizeScreen`. When `validate(_item)` returns non-null, the Save button does NOT pop; it surfaces a blocking inline error announced to screen readers (never a silent dead button). Re-scanning replaces the stored code, clearing the gate. This is targeted: a default-no-op base method leaves every other `CustomizableListItem` (themes/timers) unaffected.

Output: new `scan_register_screen.dart`, new `scan_register_card.dart`, edited `alarm_task_schemas.dart` (add the marker that mounts the card), edited `alarm_task.dart` (the `validate()` override), edited `customize_screen.dart` + `customize_list_item_screen.dart` (the save-gate hook), edited `get_setting_widget.dart` (dispatch branch), edited `dynamic_toggle_setting_card.dart` (remove the leak).
</objective>
