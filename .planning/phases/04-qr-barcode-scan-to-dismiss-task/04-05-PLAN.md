<objective>
Build the setup/registration half of the scan task: an inline "Scan to register" card (D-REG-UI) inside the scan task's `SettingGroup`, a registration scanner screen that opens only after a setup-time camera permission grant (SCAN-08), normalizes and stores the decoded code into the task's `StringSetting` (SCAN-02), shows status only ("Code registered" / "No code registered yet" — never the raw value, D-REG-DISPLAY), treats registration AS the test scan with an optional "Scan again to re-test" (D-REG-TEST / SCAN-10), deep-links to system settings on permission-denied then resumes (D-REG-CAMDENIED), and removes the pre-existing `print(setting.value)` payload leak.

Purpose: This completes the user-facing setup flow so the ring-time task (Plan 04) has a code to match. Camera permission is requested at SETUP, never at fire time (SCAN-08). Implements SCAN-02, SCAN-08, SCAN-10. The save-required gate (D-REG-REQUIRED) lives on the card's status + validation copy (`scanCodeRequired`, added in Plan 04). All ARB keys already exist (Plan 04 owns app_en.arb) — this plan does NOT touch the ARB.

**Planner decision (PATTERNS O2 / D-STORE-FORMAT):** use route (B) — a plain `StringSetting "Registered Code"` (already added in Plan 04, `isVisual:false`) PLUS a marker setting that dispatches to a custom `ScanRegisterCard` in `get_setting_widget.dart`. This honors D-STORE-FORMAT literally ("no `json_serialize.dart` factory entry needed") — the `CustomSetting` route (A) would require a `fromJsonFactories[RegisteredCode]` entry, which D-STORE-FORMAT explicitly avoids. The raw value stays in the plain `StringSetting`; the card is a non-persisted view over it.

Output: new `scan_register_screen.dart`, new `scan_register_card.dart`, edited `alarm_task_schemas.dart` (add the marker that mounts the card), edited `get_setting_widget.dart` (dispatch branch), edited `dynamic_toggle_setting_card.dart` (remove the leak).
</objective>
