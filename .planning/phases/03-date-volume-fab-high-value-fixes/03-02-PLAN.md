<objective>
Fix the rising-volume bug (VOL-01) by reimplementing the ramp independently (PR-01, sole credit per
D-PR-METHOD — no contributor attribution, implemented from scratch). Extract a pure, audio-free,
unit-testable `VolumeRampController` (a single owned `Timer` + an injected "set volume" callback + a
real `cancel()`) from the static `RingtonePlayer`, replacing the 11 fire-and-forget `Future.delayed`
callbacks and the conflated static `_stopRisingVolume` flag. Decouple "stop the ramp" from "set the
volume" so a legitimate live volume write never accidentally kills the ramp. Author the CI-runnable
`fake_async` test proving clean cancellation, no late callbacks, and no cross-alarm bleed.

Purpose: Volume is a core-value "alarm won't stop cleanly" defect. Today `setVolume()` sets the static
`_stopRisingVolume = true` on every call, so the live alarm-volume isolate port (lowering audio while
the user solves a dismiss task) silently and permanently kills the ramp; and the 11 untracked futures
bleed across alarms. The controller gives a single, cancellable, tracked ramp with one stop signal.

Output: `lib/audio/types/volume_ramp_controller.dart` (new), a rewired `RingtonePlayer`, and
`test/audio/types/volume_ramp_controller_test.dart` confirmed green via CI.
