<objective>
Fix the specific-date off-by-one (DATE-01, DATE-02) at its serialization root cause: persist a
specific date as a date-only `YYYY-MM-DD` string parsed back to a local `DateTime(y,m,d)`, normalize
the `table_calendar` picker output to a local calendar date at the source, and tolerate legacy `int`
epoch values on load (reinterpreted in UTC) so already-broken alarms self-heal. Author the CI-runnable
regression suite covering round-trip, legacy-epoch migration, and the highest-risk
`RangeAlarmSchedule` boundary.

Purpose: An alarm set for a specific calendar date must ring on exactly that day after a restart and
regardless of the device's UTC offset. The defect lives at the JSON round-trip
(`DateTimeSetting`), not in the schedule logic (`DatesAlarmSchedule.schedule()` already rebuilds from
`.year/.month/.day`), so the fix is a minimal, Tier-1 serialization + picker-boundary change.

Output: A date-only serialization format with legacy migration, a normalized picker boundary, and a
new `test/settings/types/date_time_setting_test.dart` confirmed green via CI.
</objective>
