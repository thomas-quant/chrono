<objective>
Fix the snooze state machine at its source. This plan makes every behavioral change for SNZ-01..05 in three existing files — there is NO new reschedule primitive and NO state-management library. The fix mirrors the already-correct user-list dismiss (`alarm_screen.dart:188`) and reuses the Phase-1 idempotent `updateAlarmById`/`update()` funnel unchanged.

The work is three surgical edits to one model file plus a one-line isolate wiring change (and an optional slider hardening):
1. Convert the two `.floor()` snooze-duration sites to seconds, switch `snooze()` to `clock.now()` for deterministic tests (SNZ-02).
2. Add a hard max-count gate inside `snooze()` that resolves an over-max attempt as a dismiss (SNZ-04).
3. Introduce `Future<void> _resolveDismiss()` (cancelSnooze + update, schedule-agnostic), expose it via the public async `handleDismiss()` delegator, and call it — awaited — from both the over-max snooze case and the isolate dismiss branch, replacing the incomplete synchronous `handleDismiss()` (SNZ-03/#457, SNZ-01/SNZ-05).

Purpose: Snooze does exactly what the user expects — always re-rings, honors fractional lengths to the second, enforces max count across the isolate boundary, and a snoozed-then-dismissed one-shot/dates alarm stays off for good.
Output: Modified `lib/alarm/types/alarm.dart`, `lib/alarm/logic/alarm_isolate.dart`, and `lib/alarm/data/alarm_settings_schema.dart`. No new files, no schema migration, no new dependency.

PLANNER DECISIONS (recorded here because no CONTEXT.md exists):
- D-A (over-max snooze): resolve as a DISMISS, not a no-op — research-recommended; a silent no-op risks leaving the alarm ringing forever (Research Pitfall 5). Enforced inside `snooze()`.
- D-B (SNZ-02 testability): switch `snooze()`'s time read from `DateTime.now()` to `clock.now()` (`package:clock` is already a dependency) so `withClock(Clock.fixed(...))` pins `_snoozeTime` exactly. ONLY the snooze-time write changes; do NOT touch the `DateTime.now()` comparisons inside `update()` (out of scope).
- D-C (dismiss scope): the dismiss fix calls the canonical `update()` path, NOT a once-only branch, so it is schedule-agnostic — it deactivates `OnceAlarmSchedule` (via `isDisabled`) AND finished `DatesAlarmSchedule` (via `isFinished`→`finish()`), matching the #457 generalization the reporter observed on "On Specified Days".
- D-E (test reachability): keep `handleDismiss()` as a PUBLIC `Future<void>` delegator to `_resolveDismiss()` (rather than deleting it), so the cross-file regression test in Plan 02 can drive the dismiss resolution through a public entry point without exposing the private `_resolveDismiss`.
</objective>
